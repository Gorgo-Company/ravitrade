Product Objective

A fast, elite, trader-facing options terminal.

The product is defined as a web-based options exposure terminal built around real-time options flow scanning, exposure-by-strike intelligence, magnet/gatekeeper/spring structure detection, projection modeling, multi-expiry regime analysis, contract drilldowns, historical replay, alerts, and workspace persistence.

What the product must deliver

  • Real-time options flow scanning
  • Exposure-by-strike intelligence
  • Projection path modeling
  • Historical replay and alerts

How the build thinks

Do not optimize the first month for feed ownership. Optimize for analytical ownership, execution speed, UI superiority, responsiveness, and architectural cleanliness.

Where the moat lives

The competitive edge is not raw data access. It lives in the exposure engine, strike-structure interpretation, flow scoring, projection logic, replay usability, and decision-support UX.

System Architecture

Four systems. One coherent terminal.

The implementation document defines four major layers: a data layer for ingestion and storage, a quant/inference layer for calculations, an API/streaming layer for delivery, and a frontend terminal for visualization and operator control.

01

Data Layer

Ingest and store live option trades, quotes, underlying prices, option series metadata, open interest, derived aggregates, exposure snapshots, enriched flow events, and replay history.

02

Quant / Inference Layer

Compute GEX, DEX, VEX, CEX, composite strike exposure, king nodes, gatekeepers, springs, projection paths, flow scores, and opening-versus-closing probabilities.

03

API + Streaming Layer

Expose history APIs, exposure APIs, flow APIs, contract drilldowns, workspace persistence, replay session control, WebSocket streaming, alerts, and webhook delivery.

04

Frontend Terminal

Render charts, exposure overlays, flow grid, filters, drilldown panels, multi-asset layouts, replay controls, saved layouts, and alert visualization.

Quant Engine

The actual intelligence layer.

The document defines the core notation, Greek calculations, exposure formulas, composite strike logic, expiry modes, structure rules, flow scoring, opening-versus-closing probabilities, and projection engine behavior.

Black-Scholes baseline

Use a consistent pricer for all contracts so every surface and aggregate remains internally aligned.

d1 = ( ln(S / K) + (r - q + sigma² / 2) T ) / ( sigma * sqrt(T) ) Gamma = e^(-qT) * phi(d1) / ( S * sigma * sqrt(T) ) Delta_call = e^(-qT) * N(d1) Delta_put = e^(-qT) * ( N(d1) - 1 ) Vega = S * e^(-qT) * phi(d1) * sqrt(T)

Exposure formulas

Aggregate at strike level after computing per-contract contributions.

GEX_i = Gamma_i * OI_i * M_i * S² * sign_i DEX_i = Delta_i * OI_i * M_i * S * sign_i VEX_i = Vanna_i * OI_i * M_i * S * sign_i CEX_i = Charm_i * OI_i * M_i * S * sign_i

Composite node score

Combine normalized GEX, DEX, VEX, and CEX into a single strike-level structural reading.

NET(K) = wg * z(GEX(K)) + wd * z(DEX(K)) + wv * z(VEX(K)) + wc * z(CEX(K)) Start weights: wg=.45 wd=.20 wv=.20 wc=.15 NODE_ABS(K) = |NET(K)|

Expiry modes

Support front expiry, next N expiries, and rolling DTE buckets.

0–2 DTE 3–7 DTE 8–30 DTE 31–90 DTE 90+ DTE

Structure logic

The product reads market structure through absolute node strength, path barriers, and polarity transitions.

King Node = strike with max( NODE_ABS(K) ) Gatekeeper score: B(K) = NODE_ABS(K) * Wprox(K) * Wliq(K) Path Clearance Ratio: PCR = (trend_impulse + flow_impulse) / (Bcum + eps)

Flow and projection

Use fill quality, urgency, volume context, and follow-through to build a bounded signal score and probable path target.

NBBO_POS = (P - B) / (A - B + eps) AGG = 2 * NBBO_POS - 1 FLOW_SCORE = 100 * tanh(FLOW_RAW * CONF) Target = sum( p_i * K_i )

API Contracts

Clear contracts for a fast-moving terminal.

The document defines REST endpoints for auth, symbol lookup, chart history, exposure snapshots, projections, flow scanner queries, contract drilldowns, workspaces, and replay sessions, plus WebSocket channels for live price, exposure, flow, and alerts.

POST

/api/v1/auth/login

Returns the access token and authenticated user object.

{ "email": "user@example.com", "password": "secret" }
GET

/api/v1/chart/history

Loads OHLC history for a symbol and interval range.

?symbol=SPY&interval=1m&from=...&to=...
GET

/api/v1/exposure/snapshot

Returns current spot, king node, and strike-level exposure nodes for the selected mode and regime.

?symbol=SPY&mode=next_n&n=5&regime=0_2_dte
GET

/api/v1/exposure/projection

Returns projection horizon, target, and interpolated path points.

?symbol=SPY&interval=15m&mode=front
GET

/api/v1/flow/recent

Supplies scored live flow rows, including VOI, flow score, and opening/closing probabilities.

?symbol=SPY&minScore=60&limit=100
GET

/api/v1/contracts/{occSymbol}/history

Provides OI series, trade series, and summary classification context for a single contract.

/api/v1/contracts/SPY260517C00515000/history

Critical delivery rule

Build the vertical slice in the correct order: live ingestion, normalized storage, quant inference, API and WebSocket delivery, chart and flow UI, replay and alerts, then polish and deploy.

30-Day Build Plan

A strict month-one execution sequence.

The implementation document defines a day-by-day sequence grouped into four weeks: foundation and live pipeline, quant engine and data products, terminal UI, and replay/alerts/polish/deploy.

Week 1 — Foundation + Live Pipeline

Days 1–7
  1. Day 1Initialize repo, frontend/backend/packages structure, Docker Compose, and env files.
  2. Day 2Create FastAPI shell and Next.js shell, confirm both run locally.
  3. Day 3Configure Postgres, Redis, ClickHouse, create migrations and base tables.
  4. Day 4Connect to the managed live options data provider, authenticate, subscribe to trades, quotes, and underlying prices.
  5. Day 5Build normalization layer and internal canonical schema mapping.
  6. Day 6Persist raw option trades, quotes, and underlying prices. Verify end-to-end writes.
  7. Day 7Build initial API routes and stand up the WebSocket hub.

Week 2 — Quant Engine + Data Products

Days 8–14
  1. Day 8Implement Greeks utilities and gamma exposure calculation.
  2. Day 9Aggregate by strike, implement front expiry mode and next-N mode.
  3. Day 10Implement rolling DTE buckets and composite node values.
  4. Day 11Compute king node, gatekeepers, and path clearance ratio.
  5. Day 12Detect springs, add spring scores, and trigger rules.
  6. Day 13Implement NBBO fill location and full flow score model.
  7. Day 14Implement opening-vs-closing probability, write exposure snapshots, and enriched flow events.

Week 3 — Terminal UI

Days 15–21
  1. Day 15Build terminal page shell with chart panel and layout framework.
  2. Day 16Connect chart to history API and add interval selector.
  3. Day 17Load exposure snapshot, render strike ladder or nodes, highlight king node.
  4. Day 18Add projection overlay, regime selector, and expiry mode selector.
  5. Day 19Build flow grid with live recent flow and sorting/filtering.
  6. Day 20Add row-click contract drilldown with OI and trade context.
  7. Day 21Add multi-asset layout, watchlist, and workspace save/load.

Week 4 — Replay, Alerts, Polish, Deploy

Days 22–30
  1. Day 22Build replay event reader and republish historical events over the same channels.
  2. Day 23Create replay controls: play, pause, speed selector, replay binding to chart and flow grid.
  3. Day 24Implement user alerts for price, node, spring, and flow triggers.
  4. Day 25Implement webhook delivery and summary payload generation.
  5. Day 26Optimize hot queries and cache recent snapshots, symbols, and expiries.
  6. Day 27Add auth and protect routes.
  7. Day 28Improve chart readability, spacing, typography, color logic, and filter UX.
  8. Day 29Run full integration pass across live mode, replay mode, alerts, and workspaces.
  9. Day 30Deploy production build, configure SSL, wire environment variables, and smoke-test the live terminal.

Definition of Done

What must work by launch.

  • Live options trades and quotes ingest correctly
  • Underlying prices ingest correctly
  • Normalized storage is queryable
  • Strike-level exposures are computed and streamed
  • King nodes, gatekeepers, and springs are identified
  • Flow scores and probabilities are exposed to the UI
  • Chart, overlays, grid, drilldowns, replay, alerts, and workspaces all operate in production
